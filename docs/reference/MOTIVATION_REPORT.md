## 1. The Core Problem

Software vulnerabilities are growing in volume and severity faster than human security teams can respond. The NIST National Vulnerability Database (NVD) recorded over 29,000 new CVEs in 2023 alone — roughly 80 per day — with Python-based web applications representing a rapidly increasing share of reported issues [NVD 2023]. At the same time, the widespread adoption of AI-powered code generation tools such as GitHub Copilot, ChatGPT, and Amazon CodeWhisperer has introduced a new and largely unexamined threat: these tools routinely generate code that contains security vulnerabilities, and existing detection methods are poorly equipped to catch them.

Three categories of tools exist for detecting vulnerabilities in source code: static application security testers (SAST) such as Bandit, Semgrep, and CodeQL; formal verification (FV) tools such as ESBMC and CBMC; and, most recently, large language models (LLMs) prompted to reason about security. All three categories have critical and well-documented shortcomings. SAST tools produce false positive rates of 30–60%, overwhelming developers with noise [Bennett et al. 2024]. Formal verification tools cannot detect semantic-level vulnerabilities such as SQL injection or path traversal at all [Dubniczky et al. 2025]. LLMs perform well on small, isolated code snippets but degrade significantly on functions of realistic size and complexity [Dubniczky et al. 2025].

None of these tools has been specifically trained or evaluated on real-world, CVE-confirmed Python web application vulnerabilities with per-CVE severity labels. This gap is what this project addresses.

## 2. Use Case 1 — Security of LLM-Generated Code

### The Problem

AI code generation has reached mainstream adoption at extraordinary speed. GitHub Copilot surpassed one million paid subscribers within a year of launch, and studies show that developers accept AI-generated code suggestions with minimal review [Ziegler et al. 2022]. The security implications of this are alarming.

A large-scale empirical study by Tihanyi et al. (2024), published in *Empirical Software Engineering*, found that **all examined LLMs produced vulnerable C code** when prompted to write common programming tasks [Tihanyi et al. 2024]. A subsequent study focusing specifically on Python codebases — SecureQwen — confirmed this finding for Python: LLMs consistently generate code containing CWE-89 (SQL Injection), CWE-78 (Command Injection), and CWE-22 (Path Traversal) vulnerabilities when writing web application logic [Mechri et al. 2025]. A separate analysis of 7,703 files generated by ChatGPT, Copilot, and CodeWhisperer identified **4,241 CWE instances across 77 vulnerability types**, with CWE-89 (mean CVSS 8.76) and CWE-78 (mean CVSS 8.68) being the most common [Cotroneo et al. 2024].

The core finding across all these studies is consistent: LLMs lack contextual security awareness during code generation. They produce syntactically correct, functionally plausible code that nonetheless contains the exact vulnerability classes that dominate real-world CVE disclosures.

### Why Existing Tools Fail Here

The CASTLE benchmark (Dubniczky et al. 2025) provides the most rigorous head-to-head evaluation of existing tools against these vulnerability types. Testing 13 SAST tools, 10 LLMs, and 2 formal verification tools on 250 manually crafted C programs covering 25 CWEs, CASTLE found that:

- The best-performing SAST tool (CodeQL) detected only **29% of vulnerabilities**, with SonarQube generating 2.7× as many false positives for the same recall rate.
- Formal verification tools (ESBMC, CBMC) **cannot detect SQL injection, path traversal, or command injection** at all, as these are semantic vulnerabilities that lie outside the scope of model checking.
- Even the best LLM (GPT-o3-Mini, 977 CASTLE score) missed 24 out of 150 known vulnerabilities on small, isolated, expert-crafted programs — and CASTLE itself acknowledges that LLM accuracy "declines and hallucinations increase as code size grows."

CASTLE was conducted on hand-crafted C programs averaging 42 lines. Real LLM-generated Python web functions are significantly longer, more context-dependent, and syntactically different from the training distributions these tools assume.

### How This Project Addresses It

This project builds a detection model trained exclusively on **real CVE-linked Python web code** — the same language, framework, and vulnerability classes that LLMs generate. By fine-tuning GraphCodeBERT on (vulnerable function, fix) pairs sourced from confirmed CVEs in production Django, Flask, and aiohttp repositories, the model learns the semantic patterns of real vulnerabilities rather than the toy patterns of synthetic benchmarks.

The final demo runs this model on any pasted code — including code generated by GPT-4 or Copilot — and produces a CWE classification, a CVSS severity score, and a line-level highlight of the suspicious code region. This directly answers the question CASTLE raises but cannot answer: *what happens when you apply vulnerability detection to realistic, web-framework code generated by an AI?*

## 3. Use Case 2 — CVE Triage and CVSS Severity Prediction

### The Problem

Security teams at organisations with significant software portfolios receive more CVE notifications than they can manually process. A 2024 analysis found that security engineers at mid-to-large organisations receive an average of **130+ new CVEs per day** requiring triage decisions [CaSey 2025]. Manual severity assessment — reading the NVD description, examining the affected code, and estimating exploitability — takes an experienced analyst 15–45 minutes per CVE. At scale, this means most CVEs are triaged based on the NVD text description alone, without examining the actual vulnerable code. This introduces two failure modes: high-severity vulnerabilities are deprioritised because their description is vague, and low-severity issues are over-escalated because their descriptions use alarming language.

The standard triage signal is the CVSS base score, which NVD assigns at a CVE level. However, CVSS is computed from the vulnerability description, not the code — and two CVEs of the same CWE type can have drastically different CVSS scores depending on attack vector, privilege requirements, and CIA impact profile. A SQL injection accessible without authentication over a network (CVSS 9.8) is fundamentally different from one requiring local access and low privileges (CVSS 4.3), even though both are CWE-89. Existing binary vulnerability detectors treat them identically.

### Current ML Approaches and Their Limits

Recent NLP-based approaches to automated CVSS prediction work from vulnerability text descriptions rather than code. A RoBERTa-based classifier achieves **85% accuracy on severity category prediction** from CVE text [VLAI 2024]. The CaSey system (fine-tuned GPT-3.5) achieves **73.6% combined CWE + severity accuracy** [Ramirez-Ochoa et al. 2025]. Both are text-only: they read the description and predict the score, but they never look at the vulnerable code.

This matters because the CVSS score is a function of the code's actual exploitability, not how the reporter described it. A model that reads code and predicts severity from first principles — examining whether tainted input reaches a sink without sanitization, whether the vulnerability is network-accessible, whether it affects confidentiality or integrity — would be more grounded and more generalisable.

### How This Project Addresses It

This project frames vulnerability detection as a **regression problem over CVSS scores**, not a binary classification. Every sample in the PyVulSev dataset carries a per-CVE CVSS base score fetched directly from the NVD API, along with six sub-metrics: attack vector, attack complexity, privileges required, confidentiality impact, integrity impact, and availability impact.

The GraphCodeBERT model has a severity regression head that predicts the CVSS score from code, producing not just "this is vulnerable" but "this is a 8.2/10 network-exploitable, low-complexity, high-confidentiality-impact vulnerability." This output is directly actionable for triage: security teams can rank open vulnerabilities by predicted CVSS, prioritising the most exploitable ones first without reading every CVE description.

## 4. Use Case 3 — Augmenting SAST Tools in CI/CD

### The Problem

Static analysis tools are the most widely deployed category of vulnerability detection in software development pipelines. Semgrep, Bandit, and CodeQL are integrated into CI/CD workflows at hundreds of thousands of organisations. However, their utility is fundamentally limited by the precision-recall trade-off inherent in rule-based pattern matching.

Bennett et al. (2024) conducted a systematic evaluation of SAST tools on real Python codebases and found detection rates of **11.2–26.5% for standard SAST tools**, improvable to 44.7% by augmenting Semgrep with enhanced rules — still below 50% recall [Bennett et al. 2024]. A 2026 analysis of hybrid LLM-augmented SAST found that AI filtering could reduce false positives by **up to 91%** while improving detection 2.5× — but only when the AI model was trained on domain-relevant code. Generic LLMs used as SAST filters introduce their own hallucinations, partially offsetting the gain.

The fundamental problem with rule-based SAST is that rules match syntactic patterns. A parameterised query written with a framework-specific ORM that is still vulnerable due to raw string interpolation at a lower layer will not match a Bandit rule for `cursor.execute(f"`)`. Context-dependent vulnerabilities — where the flaw only exists given the surrounding call stack — are invisible to pattern matching.

### How This Project Addresses It

A model fine-tuned on real CVE-linked code learns the semantic context of exploitable vulnerabilities, not just their surface syntax. It can serve as a second-pass filter on SAST alerts: for each Bandit or Semgrep finding, pass the flagged function to the model and use its confidence score to rank or suppress alerts. This hybrid approach — rule-based first pass, learned model second pass — combines the speed of SAST with the contextual reasoning of a fine-tuned transformer.

## 5. Use Case 4 — Automated Patch Suggestion

### The Problem and Partial Fit

Automated patch generation is an active research area with notable recent results. Google's Gemini-based system fixed **15% of sanitizer-triggered bugs** end-to-end in production [Google 2024]. VulRepair (T5-based, FSE 2022) achieves correctness on a fraction of the BigVul benchmark. The PATCHEVAL benchmark found LLMs fix **21.7% of real Python CVEs** due to poor vulnerability localization [PATCHEVAL 2024].

This use case is a secondary fit for this project. The `code_after` pairs in the PyVulSev dataset represent ground-truth developer-written fixes. These can serve as training targets or reference patches for a separate patch generation head. However, the primary contribution of this project is detection and severity prediction — patch generation is a natural extension once detection is reliable.

The attention-based line localization produced by the detection model (described in Section 8) is a prerequisite for effective patch generation: you must first identify *where* the vulnerability is before you can suggest *how* to fix it. The model's line-level output is therefore a bridge between Use Cases 1/2 and Use Case 4.

## 6. Motivating Paper: CASTLE and Its Limitations

### What CASTLE Does

Dubniczky et al. (2025) introduced **CASTLE** (CWE Automated Security Testing and Low-Level Evaluation), a benchmarking framework that evaluates 13 SAST tools, 10 LLMs, and 2 formal verification tools on 250 hand-crafted, compilable C programs covering 25 CWEs. They propose the CASTLE Score — a severity-weighted metric that penalises false positives and rewards detection of high-ranked CWEs — and conduct the most systematic head-to-head comparison of vulnerability detection methods published to date.

CASTLE's key findings:
- GPT-o3-Mini is the best single tool (977 CASTLE Score, 78% recall) but still misses 24/150 vulnerabilities on tiny, isolated programs
- The best SAST tool (CodeQL) achieves only 29% recall
- Formal verification tools cannot detect SQLi, path traversal, or command injection at all
- Tool combinations help (ESBMC + CodeQL = 814 score, +23% over ESBMC alone)
- LLM performance degrades on code longer than 400 lines

### CASTLE's Limitations and How This Project Responds

| CASTLE Limitation | Direct Quote from Paper | This Project's Response |
|---|---|---|
| Synthetic, isolated code | *"Real-world software often exhibits multi-faceted security flaws spanning tens or hundreds of files"* | All samples are real CVE-confirmed commits from production Python repositories |
| C-only; misses web vulnerability classes | *"Formal Verification methods...cannot detect...SQL injection, Path traversal"* — their best tool class fails entirely on the target CWEs | Exclusively targets CWE-89, CWE-78, CWE-22, CWE-502 in Python web frameworks |
| Binary labels only | CASTLE Score gives CWE-rank bonuses but treats all SQLi identically | Per-CVE CVSS regression with 6 sub-metrics; two SQLi bugs with different attack contexts have different scores |
| No model training | Evaluates existing tools only; concludes "GPT-o3-Mini is best" | Fine-tunes GraphCodeBERT on domain-specific data; produces a trained model, not just a tool ranking |
| LLMs degrade on large code | *"LLMs tend to report false positives when dealing with larger codebases"* | Input is the function-level diff hunk — focused context, not whole files |
| Fixed dataset enables overfitting | *"Tool vendors could theoretically fine-tune their analyzers to excel on known benchmarks"* | Red team adversarial mutations + retraining loop; the evaluation set evolves |
| No before/after pairs | Each sample is an isolated snippet with no ground truth fix | Every sample has `code_before` (vulnerable) and `code_after` (developer fix) |

The positioning statement for a paper introduction writes itself:

> *"CASTLE demonstrated that even the best existing tools — whether rule-based, formal, or LLM-based — fail on the precise vulnerability classes that dominate Python web applications, and fail even more severely as code grows toward realistic sizes. We address this directly by building a dataset and model trained on real-world, CVE-confirmed Python web vulnerabilities with per-CVE severity labels."*

## 7. What This Project Does — Full Pipeline

### Dataset: PyVulSev

**PyVulSev** (Python Vulnerability Severity Dataset) is the first Python-web-specific, multi-source vulnerability dataset with per-CVE CVSS gradient labels and before/after code pairs.

#### Data Sources (target: 1,000+ samples)

| Source | Method | Expected yield | Why |
|---|---|---|---|
| CVEfixes DB | SQL query: method-level diffs for Python CVEs | ~200 | Highest-quality source; each entry has a confirmed NVD CVE and CWE |
| OSV Schema | REST API; filter Python ecosystem, web frameworks | ~250 | Broad coverage; structured advisory format |
| GitHub Advisory DB (full clone) | Clone `github/advisory-database`; parse YAML locally | ~300–400 | Much larger than what the GHSA GraphQL API returns; same structured format as OSV |
| NVD Reference Mining | Call NVD API for Python CVEs with target CWEs; extract GitHub fix-commit URLs from `references`; fetch diffs | ~200–300 | Starts from NVD ground truth; covers CVEs that other sources miss |
| Framework Security Advisories | Scrape Django/Werkzeug/aiohttp/FastAPI security changelogs; extract commit SHAs | ~100–150 | Very high quality; single CVE per commit, documented CWE, confirmed fix |
| Snyk Python DB | Pull Snyk open-source advisories for PyPI packages with fix commits | ~150–200 | Covers smaller web libraries that CVEfixes and OSV miss |

Every sample must satisfy: confirmed CVE ID, CWE in {CWE-89, CWE-78, CWE-22, CWE-502}, Python language, non-empty `code_before` diff hunk, `syntax_valid = true`.

#### Slim Schema — 21 Fields

The schema was deliberately reduced from 46 to 21 fields. Fields retained are either model inputs, prediction targets, or minimum traceability anchors. All pipeline-internal flags, derivable values, and redundant advisory IDs have been removed to prevent the model from learning spurious correlations.

```json
{
  "id":                      "cvefixes_CWE-89_a1b2c3",
  "source":                  "cvefixes",
  "cve_id":                  "CVE-2022-28347",

  "cwe":                     "CWE-89",
  "cvss_score":              9.8,
  "cvss_severity":           "CRITICAL",

  "cvss_attack_vector":      "NETWORK",
  "cvss_attack_complexity":  "LOW",
  "cvss_privileges_required":"NONE",
  "cvss_confidentiality":    "HIGH",
  "cvss_integrity":          "HIGH",
  "cvss_availability":       "HIGH",

  "code_before":             "def get_user(username):\n    query = f\"SELECT * FROM users WHERE name = '{username}'\"\n    ...",
  "code_after":              "def get_user(username):\n    query = \"SELECT * FROM users WHERE name = %s\"\n    cursor.execute(query, (username,))\n    ...",
  "commit_message":          "Fix SQL injection in user lookup — parameterise query",

  "repo":                    "django/django",
  "fix_commit":              "abc123def456",
  "framework":               "django",

  "content_hash":            "sha256:...",
  "split":                   "train",
  "nvd_enriched":            true
}
```

**Fields dropped from the original 46:**

`_schema_version`, `scraped_at`, `ghsa_id`, `osv_id`, `pysec_id` — `cve_id` alone provides full traceability  
`cwe_name`, `vuln_type` — directly derivable from `cwe`  
`label_source`, `label_confidence` — internal pipeline state  
`cvss_version`, `cvss_vector`, `cvss_source` — `cvss_vector` is redundant with the six sub-metric fields  
`cvss_scope`, `cvss_user_interaction` — marginal signal for the four target CWEs  
`file_path`, `vulnerable_commit`, `commit_date`, `commit_author` — not used in training; `commit_author` is always null  
`language` — always `python`; zero entropy  
`loc_before`, `loc_after` — computable on the fly  
`syntax_valid`, `has_taint_source`, `is_web_code` — filtering flags used in Phase 1 cleaning only  
`pair_id` — internal dedup, superseded by `content_hash`

---

### Phase 1 — Multi-Source Ingestion

Six scrapers run in parallel and write to MongoDB Atlas via upsert on `content_hash` (prevents cross-source duplicates). Each scraper produces the 21-field schema via `build_meta()` in `src/utils/file_utils.py`. Samples with `nvd_enriched = false` are queued for Phase 3 CVSS enrichment.

Quality gates applied per sample:
- `syntax_valid`: AST parse check on `code_before`
- Minimum 5 lines of code before the fix
- CWE must be in {CWE-89, CWE-78, CWE-22, CWE-502}
- `code_before` ≠ `code_after` (non-trivial fix)

---

### Phase 2 — Dataset Split

Stratified train/val/test split (70/15/15) stratified on `cwe` × `framework` to ensure each CWE and each framework is proportionally represented across all three splits. The test set is held out entirely until final evaluation. Split assignment is written back to MongoDB as the `split` field.

---

### Phase 3 — NVD CVSS Enrichment

For samples where `nvd_enriched = false` (CVEfixes and OSV samples that do not carry CVSS at scrape time), the NVD REST API v2.0 is called with the `cve_id` to retrieve the full CVSS v3.1 record. All six sub-metric fields and `cvss_score` are populated. GHSA and Snyk samples already carry CVSS scores at scrape time and skip this phase.

---

### Phase 4 — GraphCodeBERT Dual-Task Model

**Architecture:**

```
Input: code_before (tokenised, max 512 tokens)
         ↓
GraphCodeBERT encoder (12-layer transformer with data-flow edges)
         ↓
[CLS] representation
    ├─ Head 1: CWE classifier — linear → softmax(5 classes)
    │          CWE-89 / CWE-78 / CWE-22 / CWE-502 / Other
    └─ Head 2: CVSS regressor — linear → sigmoid × 10 → [0.0, 10.0]
```

**Why GraphCodeBERT over plain CodeBERT:**
GraphCodeBERT incorporates data-flow graphs extracted from the AST during pre-training. For vulnerability detection, data flow is directly relevant: it captures whether a tainted user input (source) reaches a dangerous function call (sink) without sanitization, which is the semantic definition of CWE-89, CWE-78, and CWE-22. Plain CodeBERT treats code as a flat token sequence and misses these flow relationships.

**Training objective:**

```
L_total = CrossEntropy(CWE_pred, CWE_true) + λ · MSE(CVSS_pred, CVSS_true)
```

Where λ is a weighting hyperparameter (tuned via validation set). Optionally, a **severity-aware loss** variant penalises false negatives on high-severity samples more heavily:

```
L_severity = CrossEntropy(CWE_pred, CWE_true) × (1 + α · cvss_true / 10)
```

This ensures the model does not learn to suppress high-CVSS detections to minimise overall loss.

---

### Phase 5 — Evaluation

**Standard metrics (per CWE class):**
- Precision, Recall, F1-score
- Macro-averaged F1 across all four CWE classes

**Severity metrics:**
- RMSE and MAE on CVSS score prediction
- Severity-weighted recall: fraction of high-severity (CVSS ≥ 7.0) vulnerabilities correctly detected — penalises missing critical issues more than missing low-severity ones

**Cross-framework generalisation:**
- Train on Django-only samples; evaluate on Flask and aiohttp samples
- Tests whether learned vulnerability patterns transfer across Python web frameworks

**Baselines for comparison:**
- Bandit (rule-based SAST)
- Semgrep with default Python security rules
- CodeBERT fine-tuned (single-task binary classifier) — ablation
- GPT-4o prompted with the CASTLE system prompt (LLM baseline)

---

### Phase 6 — Red Team Adversarial Module

**Motivation:** CASTLE explicitly identifies fixed benchmarks as a path to overfitting. Vendors can tune tools to known test cases. A model that only performs well on its training distribution is not a reliable security tool.

The red team module applies **semantics-preserving code mutations** to the test set to generate adversarial examples that preserve the vulnerability but change the surface form:

| Mutation type | Example | Preserves vulnerability? |
|---|---|---|
| Variable rename | `username` → `u` or `_input_val` | Yes |
| Dead code insertion | Add unreachable `if False:` branches | Yes |
| String splitting | `f"SELECT * FROM {u}"` → `"SELECT * FROM " + u` | Yes |
| Whitespace / comment injection | Add docstrings, blank lines | Yes |
| Wrapper function extraction | Move the vulnerable call into a helper | Yes |

For each mutation, the model is re-evaluated. A drop in recall on mutated samples identifies brittleness. Adversarial samples where the model fails are added to the training set and the model is retrained. This loop runs for a fixed number of iterations (e.g., 3 rounds) and the improvement in adversarial recall is reported as a robustness metric.

---

### Phase 7 — Feedback Loop

Adversarial samples generated in Phase 6 that the model fails to detect are written back to MongoDB with `split = train` and a flag indicating adversarial origin. The model is retrained on this augmented dataset. Final evaluation compares the pre- and post-feedback model on the held-out test set and on a fresh set of adversarial mutations not seen during training.

## 8. Final Demo

### What the Demo Does

The demo is a **Gradio web application** deployed on Hugging Face Spaces (permanent public URL, free tier). It allows anyone — including professors, reviewers, and conference attendees — to paste a Python function and receive an instant vulnerability report. The primary intended use is pasting LLM-generated Python web code (e.g., a Django view generated by GPT-4 or Copilot) and seeing whether the model detects a vulnerability that standard SAST tools would miss.

### Input

A text area accepting any Python function. No framework configuration required. The user pastes raw code and clicks **Analyse**.

### Output Report

The report has four sections:

**Section 1 — Verdict**
```
Vulnerability Detected: CWE-89 — SQL Injection
Confidence: 94.2%
Severity: 8.8 / 10.0 — HIGH
```

**Section 2 — CVSS Breakdown**
A visual table of the predicted CVSS sub-metrics:
```
Attack Vector:         NETWORK
Attack Complexity:     LOW
Privileges Required:   NONE
Confidentiality:       HIGH
Integrity:             HIGH
Availability:          NONE
```

**Section 3 — Line-Level Highlight**
The pasted code is displayed with suspicious lines highlighted in red and less-suspicious high-attention lines in amber. A tooltip on each highlighted line shows the attention weight (0–100%) that contributed to the prediction.

**Section 4 — Suggested Fix**
The closest matching `code_after` sample from the training dataset is shown as a diff, giving the user a concrete example of how a developer fixed an equivalent vulnerability in a real CVE.

---

### How Line-Level Highlighting Works

GraphCodeBERT (like all transformer models) computes attention weights at every layer and every head. Each attention weight `A[i][j]` represents how much token `i` attends to token `j` when building its contextual representation. The model's prediction is a function of the `[CLS]` token's final representation, which has aggregated information from all other tokens through these attention paths.

**Step 1 — Attention rollout**

Raw attention weights at a single layer are not directly interpretable because information flows through multiple layers. Attention rollout [Abnar & Zuidema 2020] computes the cumulative attention from the `[CLS]` token to every input token across all layers by recursively multiplying attention matrices:

```
A_rollout = A_L × A_{L-1} × ... × A_1
```

Where each `A_l` is the attention matrix at layer `l`, averaged across all 12 heads. The result is a single vector of length equal to the number of input tokens, where each value represents how much the `[CLS]` token's final representation depends on that input token.

**Step 2 — Token-to-line aggregation**

The input code is tokenised by the GraphCodeBERT tokenizer (byte-pair encoding). Multiple tokens may map to the same source line. For each source line `l`, aggregate its tokens' rollout scores:

```
score(line l) = mean( rollout[t] for t in tokens(l) )
```

**Step 3 — Normalisation and threshold**

Scores are min-max normalised to [0, 1]. Lines with `score > 0.7` are highlighted red; lines with `score > 0.4` are highlighted amber. This threshold was calibrated empirically on the validation set.

**What this means in practice:**
For a SQL injection vulnerability, the model's highest-attention lines will typically be the line where unsanitised user input is interpolated into a query string — because that is the syntactic pattern most strongly associated with CWE-89 in the training data. For path traversal, attention concentrates on `os.path.join()` calls that receive user-controlled input without normalisation. The highlights therefore do not just say "this function is vulnerable" — they point to the specific line that is most likely the sink.

**Limitation to acknowledge in the paper:**
Attention rollout is an approximation. It measures what the model *attended to*, not what *caused* the prediction in a causal sense. For a more rigorous attribution, integrated gradients [Sundararajan et al. 2017] can be used as an alternative — they perturb the input and measure the gradient of the output with respect to each token, giving a causal attribution score. The demo uses attention rollout for speed (no gradient computation required at inference); the paper reports integrated gradients for the evaluation examples.

### Demo Stack

```
Gradio (UI)
  └─ HuggingFace Transformers (GraphCodeBERT inference)
       └─ Fine-tuned model weights (loaded from HuggingFace Hub)
            └─ Attention rollout (custom 50-line numpy implementation)
```

Approximate inference latency: 200–400ms per function on CPU (HuggingFace Spaces free tier). The demo is fully reproducible and the model weights will be released publicly on Hugging Face Hub alongside the paper, enabling independent replication.

### Why This Demo is Stronger Than CASTLE's Evaluation

CASTLE evaluates tools on 250 hand-crafted, expert-written, 42-line C programs. The demo of this project:
- Accepts arbitrary real-world code, including AI-generated code
- Produces a severity score, not just a binary detection
- Localises the vulnerability to specific lines
- Provides a reference fix from a real CVE
- Can be used live by reviewers, professors, and conference attendees to test cases that CASTLE's static benchmark cannot anticipate

This directly addresses CASTLE's own stated concern about fixed benchmarks enabling overfitting: the demo input is open-ended, and the model's generalisation to unseen LLM-generated Python is itself a novel empirical contribution.
