**Scope:** 13 peer-reviewed and high-quality arXiv papers, 2022–2026  
**Venues:** IEEE, ACM, USENIX, Springer, arXiv (CS.CR, CS.SE)  
**Project Context:** AI-Driven Security Analysis and Autonomous Red Teaming for LLM-Generated Flask Applications

## Summary of Trends in the Literature (2022–2025)

The literature on security vulnerabilities in LLM-generated code has grown dramatically since 2022, accelerating sharply after the public release of ChatGPT in late 2022. The field has evolved through three visible phases.

**Phase one (2022–early 2023)** was characterized by empirical measurement studies — researchers asking "how bad is this?" Studies by Khoury et al. and Allievi et al. established the empirical baseline: LLM-generated code contains real vulnerabilities at measurable rates, comparable to or worse than human developers for specific CWE types. These papers used relatively small, controlled code samples and relied on static analysis tools (CodeQL, Bandit) for vulnerability detection.

**Phase two (mid 2023–2024)** introduced systematic benchmarks and multi-agent mitigation systems. Papers like AutoSafeCoder and CWEval shifted the question from "are LLMs insecure?" to "how do we measure and fix this systematically?" This period also saw the release of major evaluation frameworks (CyberSecEval, SecurityEval, CWEval-bench) and unified datasets (DiverseVul, CVEfixes, PrimeVul). A critical finding emerged: static benchmarks were significantly overestimating model performance due to data leakage and label noise, motivating cleaner evaluation methodology.

**Phase three (2025–present)** is defined by adversarial sophistication and generalization challenges. Papers like RedCoder demonstrate multi-turn red teaming attacks that far exceed single-prompt approaches. Foundational critiques (Rethinking Evaluation, Everything You Wanted to Know) challenge whether the entire evaluation paradigm is correctly set up. The field is converging on the conclusion that vulnerability detection requires execution context, not just syntactic analysis, and that feedback loops between detection and retraining are necessary for robust models.

Across all three phases, **Python and web frameworks remain systematically underrepresented**. The dominant research substrates are C and C++, leaving Flask, Django, and Python web application vulnerabilities largely unstudied in formal academic literature.

## Key Research Gaps Justifying This Project

### Gap 1 — No Python/Flask-Specific Vulnerability Dataset
Every major vulnerability dataset (BigVul, DiverseVul, PrimeVul, LineVul training set, VulGate) is predominantly C/C++. Python appears as a minority in multi-language datasets, and Flask-specific patterns (route handlers, SQLAlchemy misuse, Jinja2 template injection, pickle in POST handlers) are not represented in any existing benchmark. This project directly fills this gap by constructing the first labeled Flask vulnerability dataset.

### Gap 2 — No Live Exploit Validation in Existing Benchmarks
Existing papers evaluate vulnerability detection statically — a scanner labels code as vulnerable or secure. No existing benchmark validates exploitability by actually attacking a running application. Our Phase 6 red team engine is the first system (in the Flask context) to close this loop: vulnerabilities are confirmed by real HTTP attacks, not just pattern matching.

### Gap 3 — Detection and Red Teaming Are Separate Tools
AutoSafeCoder detects vulnerabilities. RedCoder generates attacks. CyberSecEval evaluates models. These are three separate research silos. This project integrates all three into a single continuous pipeline where red team failure rates directly trigger model retraining — a closed loop that no prior work implements end-to-end for Python web applications.

### Gap 4 — Insecure Deserialization and Path Traversal Are Neglected
The survey by Basic & Giaretta explicitly identifies CWE-502 (Insecure Deserialization) and CWE-22 (Path Traversal) as the most neglected vulnerability categories in Python security research. SQL Injection and XSS receive the most attention. This project gives equal treatment to all four categories, with dedicated detection rules, payload generators, and labeled samples for each.

### Gap 5 — No Self-Improving Feedback Loop
All existing systems are static: trained once, evaluated once. Papers like Rethinking Evaluation and Data and Context Matter demonstrate that static models fail catastrophically on distribution shift (new codebases, new vulnerability patterns). Our Phase 7 feedback loop — where red team exploit results augment the training dataset and retrigger model training — creates the first self-improving vulnerability detection system for LLM-generated Flask code.
